# 后端需求草案

## 1. 当前目标

当前后端需求还没有完全确定，本文件用于记录暂定想法，后续根据组员讨论结果修改。

## 2. 可能需要的功能

后端可能需要支持：

* 查询当前系统状态。
* 接收 UI 启动任务的指令。
* 接收 UI 停止任务的指令。
* 接收 UI 复位系统的指令。
* 接收 UI 选择任务类型的指令。
* 返回任务进度。
* 返回运行日志。
* 模拟小车和机械臂执行流程。
* 为后续真实硬件控制预留接口。

## 3. 可能的状态设计

暂定状态包括：

* IDLE：空闲。
* MOVING：小车移动中。
* READY：设备到位，准备执行。
* PICKING：采摘中。
* PLACING：放置中。
* DONE：任务完成。
* STOPPED：任务停止。
* ERROR：异常。

这些状态只是草案，后续可以根据真实流程调整。

## 4. 可能的接口设计

以下接口只是草案，不是最终设计：

* GET /status：查询状态。
* POST /start_task：启动任务。
* POST /stop：停止任务。
* POST /reset：复位。
* POST /select_task：选择任务类型。

后续需要根据 UI 和设备控制需求确认接口名称、参数和返回格式。

## 5. 暂时不做的内容

当前暂时不做：

* 不直接控制真实机械臂。
* 不直接控制真实小车。
* 不直接写复杂 ROS 通信。
* 不做数据库。
* 不做用户登录。
* 不做复杂前端页面。

## 6. 后续需要向组员确认的问题

1. UI 需要哪些按钮？
2. UI 需要显示哪些状态？
3. 小车是否已经有控制接口？
4. 机械臂是否已经有控制接口？
5. 后端需要连接 ROS，还是只需要 HTTP/TCP？
6. 任务流程是“小车先到位再机械臂采摘”，还是后端只负责机械臂？
7. 是否需要区分红苹果和绿苹果？
8. 是否需要保存实验记录？
9. 是否需要展示错误信息？
10. 最终演示时，后端需要真实控制设备，还是只展示通信和状态流程？
